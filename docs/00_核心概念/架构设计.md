# 拟态架构设计

> v1.0.0 | 核心范式：角色拥有世界坐标，窗口跟随移动

## 核心设计理念

### 世界坐标系统

拟态 v1.0.0 的核心范式转换：角色不再是禁锢在静态窗口中的像素画，而是拥有**屏幕世界坐标**的虚拟生命。

```
v1.0 前:  Window → Character inside (static)
v1.0.0:   Character has world position → Window follows (dynamic)
```

- `worldX` / `worldY`：角色中心在屏幕坐标系中的位置
- 角色始终绘制在窗口中央（`winW/2`, `winH/2`）
- 窗口实时跟随角色移动：`window.setPosition(worldX - winW/2, worldY - winH/2)`
- 整个桌面都是角色的领地

## FSM 状态机设计

### 核心 API

```javascript
M.FSM.transitionTo('stateName', { context })  // 切换状态
M.FSM.update(performance.now())               // 每帧更新
M.FSM.getExpression()                         // 获取当前表情名
```

### 生命周期

```
_exit(oldState) → _enter(newState) → _update(newState, now) [每帧]
```

## 召唤系统设计

三通道触发，统一使用 `M.Walk.walkTo(screenX, screenY, speed, onArrive)`：

```
通道1: 右键菜单 → "召唤到这里"
  → main process: createSummonOverlay()
  → 全屏透明 BrowserWindow 出现
  → 用户点击 → IPC 'summon-target' → destroySummonOverlay()
  → renderer: walkTo(screenX, screenY, 'run')

通道2: 三击角色
  → click.js 计数 clickCount >= 3
  → M.ipc.send('start-summon') → 同上流程

通道3: 文字指令
  → 按 Enter → 输入 "过来"/"召唤"/"summon"
  → M.ipc.send('start-summon') → 同上流程
```

全屏召唤覆盖层是**临时**的：用完即销毁，不占用常驻资源。

## 自主 AI 设计

```
每 3 秒决策一次（仅在 idle 状态）:
┌─────────────────────────────────────────┐
│  roll = Math.random()                   │
│                                         │
│  roll < 0.35 ──→ startWander()          │
│    · 随机屏幕坐标                        │
│    · walkTo(target, 'walk')             │
│    · 到达后显示 curious → 回 idle       │
│                                         │
│  roll < 0.50 ──→ doMicroAction()        │
│    · 伸懒腰（举臂 800ms）                 │
│    · 歪头（随机角度 1200ms）              │
│    · 小跳（弹跳 300ms）                   │
│                                         │
│  else ────────→ 继续 idle（什么都不做）    │
└─────────────────────────────────────────┘

持续监控:
  · 鼠标靠近 150-500px → 30% 概率走过去看
  · 60s 无交互 → sleepy 状态
  · 鼠标活动 → 立即唤醒
```

## 生命感系统

三级叠加，始终运行（`animations.js` 的 `tickAnim()` 中）：

| 层级 | 效果 | 周期 | 振幅 |
|------|------|------|------|
| 底层 | 呼吸起伏 | ~3s 正弦波 | 缩放 ±2% |
| 中层 | 随机眨眼 | 2-6s 间隔 | 持续 100ms |
| 上层 | 微晃动 | ~5s 正弦波 | 水平 ±0.3 格 |

偶发动作由 AI 控制器的 `doMicroAction()` 触发。

## 关键设计决策

| 决策 | 理由 | 日期 |
|------|------|------|
| 世界坐标 + 小窗口移动 | 避免全屏覆盖层的性能开销；参考 Neko.js / Claude Pet 模式 | 2026-06 |
| Canvas 2D 而非 WebGL | 16×20 像素网格不需要 GPU 渲染；像素控制更精确 | 2026-06 |
| FSM 而非行为树 | 状态数有限（~6个）；逻辑直观清晰 | 2026-06 |
| 层次化 AI（决策层 + FSM） | 解耦自主行为和状态管理 | 2026-06 |
| damp() 帧率无关缓动 | 不同显示器刷新率下移动速度一致 | 2026-06 |
| 临时召唤覆盖层 | 用完即销毁，不常驻占资源 | 2026-06 |
| 全局命名空间而非 ES Module | Electron 渲染进程直接加载 script，无打包工具 | 2026-06 |
| IPC 扁平参数 | 绕过上下文隔离的对象序列化边缘情况 | 2026-06 |

## 数据流

```
用户拖放文件 → drop.js 递归复制 → working 状态 + 进度气泡
                ↓ 完成
           POST /api/file/process → 晓风后端处理

用户输入文字 → chat.js
                ├── 闲逛/召唤关键词 → AI/Walk 操作
                ├── 听歌关键词 → 耳机 + happy 状态
                ├── apiEnabled=true → POST /api/chat → 晓风回复
                └── apiEnabled=false → Mock 关键词匹配

音乐检测 (3s轮询) → tasklist 扫描进程
                ├── 匹配到播放器 → music-playing IPC → 耳机 + happy
                └── 播放器消失 → music-stopped IPC → 回到 idle
```
